# 의사결정 문서 — Mini Channel Talk

> 채널톡 FDE 사전 과제 — 구현한 기능, 선택 이유, 구현 순서, 의도적으로 제외한 기능, 받아들인 trade-off, 더 시간이 있었다면 할 것, 실제 투입 시간을 정리한 문서입니다.

## 1. 채널톡 코어에 대한 해석

채널톡은 메신저 · 팀챗 · CRM · 워크플로우 · ALF · 도큐먼트 · 마케팅까지 영역이 넓습니다. 그래서 코어를 한 기능으로 좁히지 않고, **그 모든 기능이 그 위에서 작동할 수 있는 받침이 무엇인지**를 먼저 정의했습니다.

다른 제품과 채널톡이 갈리는 지점은 아래와 같습니다:

- 무료 플랜 + 가벼운 위젯 → SMB가 첫날부터 채택 가능
- 익명 방문자 추적 → 식별 전에도 컨텍스트가 누적됨
- 식별 시점의 자연스러운 history merge → 익명 → 식별로 넘어갈 때 과거 흐름이 끊기지 않음

이 메커니즘이 SMB 고객사가 채널톡 위에서 컨텍스트를 처음부터 쌓을 수 있게 만들고, 그 위에 ALF나 워크플로우 같은 자동화가 얹히는 구조라고 판단했습니다. 

그래서 제가  미니 채널톡의 한 줄 정의는:

> **“익명의 방문자가 위젯에서 말을 거는 순간부터, 그 사람의 정체성·맥락·과거 대화가 운영자 화면 한 곳에 모이는 경험을 끝까지 보여준다.”**

## 2. 구현한 기능 — 선택 이유와 순서

> 구현 우선순위는 위 한 줄 정의에 대한 기여도 순으로 선정했습니다. P0는 필수적인 동작 기능, P1은 P0를 바탕으로 한 디테일, P2는 부가적인 기능입니다.

### P0

| ID | 기능 | 코어 해석과의 연결 |
|---|---|---|
| F1 | 양방향 실시간 메시징 + 대화 영속성 | 표면 = 메신저. 가장 먼저 작동해야 하는 부분. |
| F2 | 운영자 인박스 (대기 / 응대중 / 종료 + 다중 대화 동시 보기) | 운영자가 컨텍스트를 보는 자리. |
| F3 | 방문자 식별과 컨텍스트 자동 수집 (쿠키 익명 ID, URL/referrer/방문 횟수, 식별 시 자연스러운 머지) | **코어 해석의 가장 직접적 증거.** |
| F4 | 고객 프로필 패널 (재방문 시 과거 대화 자동 부착) | 컨텍스트 누적의 시각적 결정타. |

### P1

| ID | 기능 |
|---|---|
| F5 | 대화 상태 머신 (대기 → 응대중 → 종료, customer 메시지로 자동 reopen) |
| F6 | 운영자 온라인/오프라인 표시 (heartbeat) |
| F7 | 메시지 읽음 표시 |
| F8 | 전송 중 상태 (낙관적 UI) |
| Polish | unread 뱃지, 사이드바 정보 패널, 타이핑 인디케이터 (Realtime Broadcast) |

### P2

| ID | 기능 |
|---|---|
| F9 | 운영자 전용 내부 메모 (인터널 노트) |
| F12 | KPI 미니 대시보드 (운영자 콘솔 상단 6장) |
| W | 고객측 워크플로우 트리 (분기형 셀프 해결, “도움이 됐어요” 종료) |

## 3. 의도적으로 제외한 기능

여러 기능 후보 중 **위의 한 줄 정의에 직접 기여하지 않아 검증 범위가 분산되는 기능**이라고 판단되어 제외시켰습니다.

| 제외 | 근거 |
|---|---|
| 팀챗 풀버전 | 운영자 ↔ 운영자 협업 표면. 고객 ↔ 운영자 접점이 코어 |
| 모바일 앱 / 네이티브 SDK | 데스크톱 데모만으로 한 줄 정의 시연 충분. SDK 작업은 ROI가 매우 낮음. |
| 멀티 워크스페이스 | 단일 워크스페이스 가정으로 충분히 시연 가능 |
| 파일 업로드 | 우선 텍스트 기반 집중으로 생략 |
| 멀티 채널 어댑터 (인스타·라인·이메일·카카오) | customer-as-root 모델로 인박스 추상화는 이미 되어 있지만, 핵심은 아니라고 판단 |
| 전화 / Meet | 위에서 정의한 코어와 다른 차원의 부가가치 |

## 4. 트레이드오프

### 4.1 인프라 / 스택

| 영역 | 미니에서의 선택 | 프로덕션이라면 |
|---|---|---|
| Realtime | Supabase Realtime (postgres_changes / Broadcast) | WebSocket 자체 구현 + 재연결 / 백프레셔 / 메시지 순서 보장 |
| DB | Postgres 단일, 캐시·큐 없음 | Read replica + Redis + 메시지 큐 |
| 매니저 인증 | `?as=demo-manager` query param 자동 로그인 | OAuth + 권한 모델 |
| 메시지 순서 | timestamp + sequence 컬럼 | Lamport clock / 분산 시퀀서 |
| 테넌시 | 단일 워크스페이스 가정 | 멀티 워크스페이스 + 격리 |

인프라 구현 능력이 본 과제의 핵심은 아니라고 판단했기에 서드파티 프레임워크를 최대한 활용했습니다.

### 4.2 구현 단순화 (시간 절약 경로)

- **재인식률 측정 — 30분 inactivity 게이트.** Spec 초안은 “매 init = visit_count +1”이었지만 그러면 새로고침 한 번이 +1로 잡혀 KPI가 부풀려집니다. Intercom·GA4 표준대로 30분 timeout을 적용해서 *진짜 재방문*만 카운트하도록 했습니다.
- **closed conversation reopen 정책.** 닫힌 대화에서도 customer 입력을 살려두고, customer가 새 메시지를 보내면 같은 conversation이 자동 reopen + reopened_count++됩니다. 채널톡 본 제품의 customer-side thread continuity와 정합하고, 재오픈률 KPI가 의미 있는 숫자가 됩니다.
- **고객측 워크플로우 트리 = 클라이언트-only 분기.** 분기 메시지를 모두 DB에 저장하면 운영자 인박스에 불필요한 문의가 누적된다고 판단했습니다. FAQ 워크플로우 트리 분기 진행은 클라이언트-only로 하고, “상담원에게 직접 문의” 시점에만 path summary를 대화에 전송하도록 했습니다.

## 5. 실제 투입 시간

총 **약 15–20시간**,

| 구간 | 시간 | 주요 활동 |
|---|---|---|
| 목요일 저녁 | 3–4h | 채널톡 기능 분석, 코어 해석 정리, 빌드 우선순위(P0/P1/P2/명시적 제외) 확정, AI 활용 정책 정의, 데이터 모델 초안 |
| 금요일 저녁 | 4–5h | P0 — Supabase 클라이언트, visitor init, 양방향 메시징, 식별 폼, 프로필 부착, 재방문 인식 |
| 토요일 오전 | 2–3h | P1 — 대화 상태 머신, 자동 reopen, 매니저 heartbeat, 읽음 표시, 낙관적 UI, 타이핑 인디케이터 |
| 토요일 오후 | 4–5h | P2 + 데모 인프라 — 인터널 노트, KPI 대시보드, 데모 사이트, 매니저 인증 우회, 시드 데이터, 워크플로우 트리, 셀프 해결률 KPI |
| 토요일 저녁 | 2–3h | 랜딩 페이지, 데모 가이드, 제출 문서(의사결정·KPI 시트·아키텍처) 정리, 자체 검증, 제출 |
